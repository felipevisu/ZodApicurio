The cultural problem

Today

The data team pulls. Reads tables, runs CDC, scrapes logs, builds pipelines on top of schemas it doesn't control.

When it changes

Pipeline breaks. No warning. No versioning. No clear owner.

Inverted culture

Whoever produces the data should deliver it, not be hunted down.

The inversion

The data team defines the contract.
Producers deliver inside the contract.
  • Producers own responsibility for the format.
  • Consumers get guarantees about what arrives.
  • Changes become pull requests — not surprises in production.

What is Apicurio?

Open-source schema registry maintained by Red Hat.
Single source of truth for API and event contracts.

FORMATS

Multi-schema

JSON Schema, Avro, Protobuf, OpenAPI, AsyncAPI, GraphQL.

VERSIONING

Full history

Every artifact has immutable versions + labels + metadata.

RULES

Native compat

BACKWARD, FORWARD, FULL, NONE — enforced at publish time.

INTEGRATION

REST + UI + SerDes

HTTP API v3, web console, libs for Kafka producers/consumers.

What is Zod?

TypeScript-first validation library.
Single schema → inferred TS type + runtime validator.

define schema


import { z } from "zod";

const Refund = z.object({
  id: z.string().uuid(),
  amount: z.number().positive(),
  reason: z.enum(["fraud", "duplicate", "other"]),
  refundedAt: z.string().datetime()
});

type Refund = z.infer<typeof Refund>;
// ↑ TS type generated for free

parse at runtime


const result = Refund.safeParse(input);

if (!result.success) {
  result.error.issues;
  // [{ path: ["amount"],
  //    message: "...",
  //    code: "too_small" }]
}

result.data; // typed as Refund

Composable: z.object, z.array, z.union, .refine, .transform. Zero deps. Tree-shakeable.

The Zod Data Service

  • Boots up empty. Starts polling Apicurio.
  • For each version: jsonSchemaToZod(schema) → Zod code as a string → live validator.
  • Registers a single dynamic route: POST /:groupId/:artifactId/v:version
  • OpenAPI spec rebuilt on every sync, served at /openapi.json — Swagger updates itself.

Honest tradeoffs

What you gain

  • Pipeline can't break on bad data. Invalid payload → 400 at the boundary; Kafka and DB only ever see what the schema accepted.
  • Producers own delivery. No more scraping logs or chasing tables — whoever creates the data POSTs it, inside the contract they signed off on.
  • Endpoint + validator generated from the schema. Merge a JSON Schema → route, Zod check, OpenAPI entry appear. Nothing hand-written, nothing to drift.
  • Breaks fail in CI, not in prod. Apicurio dry-run on the PR; branch protection holds the line before the consumer ever sees it.

What you pay

  • Extra hop between producer and pipeline. Net-new service to run, deploy, monitor, and bill for. Adds a network jump and a place that can be down.
  • Registry becomes load-bearing. If Apicurio is wrong, stale, or unavailable, every consumer is wrong with it. Needs HA, backups, and a locked write path — CI-only, never humans.
  • Validator scales with traffic, not schema count. Every event goes through Zod before the pipeline. Throughput ceiling = validator throughput. Horizontal scale and back-pressure planning are now your problem.
  • BACKWARD checks shape, not meaning. Required field added → blocked. Enum value reinterpreted → invisible. Schema gate ≠ semantic contract; versioning discipline is still on you.
